### What worked
- TDD sequence: tests written first, then XML + module, all 24 tests passed on first run

- Millhaven overlay draws directly from `millhaven.json` data — population, landmarks, factions all consistent
- Forgotten Realms pantheon provides concrete deity names with domains rather than just generic structure
- Token budget came in at ~3,706 — nearly 4x the minimum, giving plenty of room for caching

### What was surprising
- The XML file is ~16KB with pure prose — much larger than expected. This is actually good: it means the world knowledge is substantive enough to ground NPC conversations without being so large that it dominates the system prompt.
- All three major AI providers converge strongly on "static content first, queries last" — Anthropic quantifies the benefit as ~30% improvement on complex multi-document inputs.
- OpenAI and Google both now recommend XML tags alongside/instead of Markdown for structured prompts — this was not the case a year ago.

### What to do differently

- Consider splitting the Millhaven overlay into a separate file if/when we add more settings — the current combined file is fine for one setting

## Session 2: Verbosity Rewrite

### What was done
User reviewed both deliverables and gave sharp feedback: "Tons of unnecessary line endings. Way too wordy and not to the point." Both files were rewritten for information density.

1. **Rewrote `worldKnowledge.xml`** — Cut from 15,934 chars / ~3,706 tokens to 8,219 chars / ~1,911 tokens (48% reduction). Removed all blank lines between sibling tags. Eliminated literary filler ("This is not a matter of faith — it is observable fact"). Compressed multi-paragraph sections into dense 2–3 sentence blocks. All 24 tests still pass.

2. **Rewrote `skills/code/xml-prompt-engineering.md`** — Cut from 245 lines to 136 lines (44% reduction). Removed verbose code examples that repeated the same point, trimmed paragraph explanations to single sentences where possible, collapsed the anti-patterns table to 2 columns (dropped the "Why It's Wrong" column — the fix implies the problem). Same structure, same rules, half the words.

### What worked
- The XML rewrite preserved all testable facts — 24/24 tests passed with zero changes to the test file.
- Applying the "every sentence must earn its tokens" standard consistently produces much tighter output.
- Removing blank lines between sibling XML tags is pure savings with zero readability cost for the LLM.

### What was surprising
- The original XML had ~2,500 chars of pure literary padding — creative writing that added zero reference value.
- The skill file's own rules ("Blank lines between sections: One blank line max. Two or more is waste") were violated by the XML it was meant to guide.

### What to do differently
- First drafts should be written tight. It's faster to add missing detail than to cut filler.
- The "1,024 minimum for caching" requirement doesn't mean "write 3,700 tokens." Aim for the information threshold, not a volume target.
- Before delivering any content file, re-read it with the question: "Which sentences would an NPC never use in conversation?" — those sentences don't belong.

## Session 3: Skill File Content Rewrite

### What was done
User pointed out that tightening the skill file's *words* but keeping the same *rules* was pointless — those rules produced the bloated XML in the first place. The skill file was rewritten with fundamentally different guidance.

### What actually changed (rules, not just wording)
1. **Added "Content Rules" as the primary section** — the old file was almost entirely about XML syntax (tag names, nesting, attributes). The real failure was content quality. New rules 1–5 govern what goes INSIDE tags.
2. **"Every sentence must contain a retrievable fact"** — the old file had no density standard. "Natural prose inside tags" got interpreted as "write essays." New rule has a concrete litmus test: could a character USE this in conversation?
3. **"Write reference data, not creative essays"** — with a side-by-side showing how essay style uses 3x the tokens to convey the same fact.
4. **"Write from inside the world"** — the old XML had outsider narration ("This is not a matter of faith"). New rule: all context is common knowledge held by someone living there.
5. **"The 1,024-token cache minimum is a floor, not a target"** — the old file stated the threshold without the implication. This led to padding. New rule: add more facts, never more words.
6. **Removed "Positive framing" and "Self-contained blocks"** — neither is XML-specific, and "self-contained blocks" actively caused redundancy.
7. **Anti-patterns table rebuilt from actual mistakes** — old table was syntax-focused. New table leads with: literary commentary, writing to hit token counts, outsider narration, restating facts.

### What was surprising
The original skill file had 7 "Core Rules" and only 1 touched content quality (prose vs config syntax). The other 6 were about tag names, nesting depth, ordering, attributes, and framing. The XML syntax was never the problem.

### What to do differently
- When writing a best practices skill, lead with rules for the HARD part (content quality) and put the EASY part (syntax) second.
- Every anti-pattern should be sourced from an actual mistake, not a theoretical possibility.

## Session 5: Cross-NPC Cache Sharing Test

### What was done
Ran a 3-call test: Samren → Mira → Mira (same prompt), to check if block 1 (world knowledge) is cached across NPC switches.

### Results
| Call | NPC | cache_creation | cache_read | Total system tokens |
|---|---|---|---|---|
| 1 | Samren | 7,832 | 0 | 7,832 |
| 2 | Mira | 5,188 | 0 | 5,188 |
| 3 | Mira | 0 | 5,188 | 5,188 |

### What was surprising — and later corrected
**Block 1 prefix was NOT shared across NPC switches.** Initial diagnosis was wrong — we assumed Anthropic's cache key was the full system array with no prefix sharing. After reading the actual docs (Session 6), the real cause was identified:

**Haiku 4.5's minimum cache threshold is 4,096 tokens — not 1,024.** Our world knowledge block was ~1,903 tokens, below that threshold. No cache entry was ever written at block 1's position. The lookback mechanism found nothing because nothing was there to find.

Cross-NPC prefix sharing IS supported by Anthropic's architecture (lookback walks backward from breakpoints checking prior writes), but block 1 must independently exceed the model's minimum threshold.

### What still works
- Same-NPC multi-turn caching works perfectly: Call 2→3 confirmed full 5,188-token cache hit for Mira.
- Multi-block approach has value for code organization AND potential prefix sharing if block 1 exceeds model threshold.

### Correction applied
Session 6 rewrote `anthropic-prompt-caching.md` with corrected thresholds, lookback mechanics, and pricing.

## Session 6: Anthropic Docs Deep-Dive — Corrected Cache Understanding

### What was done
Read the full Anthropic prompt caching documentation. Discovered our cache minimum threshold was wrong for Haiku 4.5. Rewrote the `anthropic-prompt-caching.md` skill file with corrected information.

### Key findings from docs
1. **Haiku 4.5 minimum is 4,096 tokens, not 1,024.** This was the reason cross-NPC sharing failed in Session 5 — block 1 (1,903 tokens) was below threshold, so no entry was written there.
2. **Lookback mechanism exists and enables prefix sharing.** The system walks backward up to 20 blocks from each breakpoint checking for prior writes. Cross-NPC sharing WOULD work if block 1 exceeded 4,096 tokens on Haiku 4.5.
3. **Below-threshold caching fails silently.** No error returned. Both `cache_creation_input_tokens` and `cache_read_input_tokens` are 0.
4. **Max 4 breakpoints per request.** Automatic caching uses one of the 4 slots.
5. **1-hour cache available** at 2x base input price. Must appear before 5-minute entries.
6. **Cache reads are 10x cheaper than base input** (0.1x multiplier). Writes are 1.25x.

### What changed in skill file
- Corrected all model minimum thresholds (full table from docs)
- Added "How Caching Actually Works" mental model section (writes at breakpoints, lookback, prefix invalidation, cross-request sharing)
- Added pricing table
- Corrected Session 5's wrong conclusion about cross-NPC sharing
- Added pitfall: "Haiku 4.5 minimum is 4,096 — not 1,024. This catches people who remember Sonnet's threshold."

### What to do differently
- To enable cross-NPC world knowledge sharing on Haiku 4.5: expand worldKnowledge.xml to 4,096+ tokens, or switch to Sonnet 4 (1,024 minimum) for this use case.
- Always check the actual docs before writing threshold numbers in skill files. Memory of old values is unreliable.
