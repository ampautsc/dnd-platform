## Session 3: Full NPC Name Path Trace — Identifying Display Name Leaks

### What Was Done
Traced the full path NPC names take from content data → DM engine → API routes → client rendering for both the encounter path (EncounterSessionService) and scene path (SceneEngine).

### Key Findings
1. **RelationshipRepository.getDisplayName() is only wired into NPC consciousness prompts** — it gates what the *NPC* perceives, not what the *player* sees.
2. **All player-facing fields use raw real names from content package** — `personality.name` / `participant.name` are used directly for:
   - `npcResponses[].senderName` (encounters)
   - `npcActions[].participantName` (scenes)
   - `transcript[].participantName` (scenes)
   - `npcs[].name` / `participants[].name` (both)
3. **API routes and controllers are pure passthroughs** — no name transformation happens between DM engine and client.
4. **SceneNarrator receives real names** and passes them to the LLM, which then uses them in narration text.
5. **The encounter path (EncounterSessionService.sendMessage) doesn't use RelationshipRepository at all** for name gating — no `nameResolver`, no `getDisplayName()` calls on the response path.

### Where Interception Must Happen
- Scene path: `SceneEngine.submitAction()` and `advanceNpcTurns()` where `npcActions` array and transcript entries are built, and where participants are serialized via `toJSON()`
- Encounter path: `EncounterSessionService.sendMessage()` where `senderName` is set, and `createEncounter()`/`getEncounter()` where `npcs[].name` is returned
- SceneNarrator: `narrateNpcBatch()` and `narrateSceneOpening()` where names are fed to the LLM
- SceneController: `createAtLocation()` where participant names are set from personality data

## Session 10: Empty Transcript — Root Cause, Missing Tests, and UX Fix

### What Was Done

1. **Found the missing test** — the `POST /api/scenes/at-location` route had 6 API integration tests asserting `id`, `status`, `participants`, `worldContext`, `openingActions` — but ZERO tests asserting that `transcript` was non-empty and contained a DM opening narration entry. 2. **Confirmed the API is correct** — the new test passes (and a direct `Invoke-RestMethod` confirmed `transcript count: 1` from the live server).

3. **Fixed the UX bug** — `handleEnterLocation` was setting `sceneProcessing = false` before firing the fetch. During the 5-30 second LLM call, there was no loading indicator at all. The user saw the static "Setting the scene…" placeholder with nothing moving and assumed it was the final answer. Fixed by:
   - Setting `sceneProcessing = true` BEFORE the fetch fires
   - Setting `sceneProcessing = false` in both `.then()` and `.catch()`
   - Changing the empty-transcript placeholder to only show when `!processing` (suppressed during load)
   - The `processing` indicator now reads "Setting the scene…" when `transcript.length === 0` (initial load) and "NPCs are thinking…" when transcript has entries (mid-scene NPC resolution)

4. **Fixed "You pushes..." grammar** — when `playerName = 'You'` (the default from `handleEnterLocation`), the LLM prompt had "You pushes open the door and steps inside" — grammatically wrong. Added an `arrivalVerb` check: `playerName.toLowerCase() === 'you'` → "push open the door and step", otherwise "pushes open the door and steps".

5. **Updated NpcScene.test.jsx** — `shows thinking indicator when processing is true` test was using `mockScene` (empty transcript) — after the conditional text change it would show "Setting the scene…" not "NPCs are thinking…". Updated to use `sceneWithTranscript`. Added new test `shows setting the scene indicator during initial load (processing=true, empty transcript)`.

**Result: 2836 tests passing**

### What Worked
The API integration test confirmed the transcript was there all along — the problem was purely UX. The user was seeing a static placeholder for 10-30 seconds (LLM processing) with no animation or movement to indicate loading was happening, and concluded it was the final state.

### What Was Surprising
The hardest part of this bug was that it was untestable from the API side — the API was always correct. The bug was purely in the client state management. Which is exactly why client state tests matter too.

### The Lesson (AND IT'S IMPORTANT)
**If you change a placeholder text but don't add a test for what the loading state LOOKS LIKE, you haven't fixed anything.** The test should have been: "while loading (processing=true, transcript=[]), a loading indicator is visible." That test would have immediately failed if `processing` was never set to `true` during `handleEnterLocation`.

Test the BEHAVIOR of the loading state, not just the happy path.

- Consider adding a spinner animation to the "Setting the scene…" text (CSS keyframe animation) so it's visually obvious it's loading
- Consider adding a timeout (e.g., 45 seconds) after which the client retries or shows an error message

## Session 12: Narrator Anti-Repetition Directive — Prompt-Level Fix

### What Was Done

**Problem**: Even with conversation history threading (Session 11), the narrator still repeated scene-setting language ("woodsmoke", "spilled ale", tavern entrance description) on follow-up narrations. The history fix was necessary but not sufficient — the system prompt still dumps the full atmospheric palette with "Use this atmosphere. Weave it into transitions, pauses, and beats" on every call, and the user message says "Narrate what the adventurer perceives" which invites re-description.

**Root Cause**: Two-layer problem:
1. Conversation history was confirmed working (Call 4 now has user/assistant/user), BUT
2. The follow-up user message prompt had no directive telling the LLM not to re-describe what it already narrated. The system prompt's `[THE WORLD RIGHT NOW]` section actively encourages atmospheric weaving.

**Fix — prompt-level anti-repetition directive:**
- When `priorHistory.length > 0` (scene already opened), the `narrateNpcBatch` user message now appends: "IMPORTANT: The scene has already been set. Do NOT re-describe the setting, atmosphere, or environment. Do not repeat sensory details (smells, lighting, sounds) that were established in previous narration. Focus ONLY on the new actions, dialogue, and character reactions."
- When no prior history exists (first call, or no sceneId), the directive is omitted — backward compatible.
- 2 new tests: one verifying the directive IS present with history, one verifying it is NOT present without history
- Fixed a `ReferenceError` from using `priorHistory` before its `const` declaration — moved the declaration above the `userMessage` construction

**Test Results**: 2848/2849 pass across the full monorepo (66/66 SceneNarrator, 49/49 SceneEngine). The 1 failure is same pre-existing PromptContent.test.js issue.

**Inspect script verification**: Re-ran `inspect-mira-prompt.mjs` — Call 3 and Call 6 (both follow-up narrator calls) now show the anti-repetition directive in the user message AND the full conversation history from prior calls.

### What Worked
- Running the inspect script to see EXACTLY what the LLM receives — proved the history was threading but the prompt was still inviting repetition
- Conditional prompt injection: only add the anti-repetition directive when history exists

### What Was Surprising
- History threading alone wasn't enough — the LLM follows the system prompt's "weave in atmosphere" instruction literally, even when it can see it already did that
- Variable ordering matters with `const` in JS — `priorHistory` was used before declaration, causing 34 test failures

### What To Do Differently Next Time
- When the LLM repeats itself, check BOTH the message history AND the prompt instructions — the prompt may be actively encouraging the behavior
- Always verify variable declaration order when refactoring — `const` has temporal dead zone
