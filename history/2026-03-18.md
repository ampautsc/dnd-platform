## Session 2: Deep Infrastructure Research for RelationshipRepository

### What Was Done
Comprehensive read of all 9 files involved in memory/relationship infrastructure:
- EncounterMemoryService (328 lines) — per-session in-memory trust/emotion/secrets
- PersonalityEvolutionService (326 lines) — cross-session permanent evolution, has `crystallizeEncounter()` but it's never called automatically
- NpcRuntimeContext (117 lines) — location/activity/mood/day experiences
- EncounterSessionService (449 lines) — encounter orchestration, `endEncounter()` does nothing except set status
- SceneEngine (664 lines) — initiative-based scenes, `endScene()` does nothing except set end reason
- SceneNarrator (185 lines) — DM narration layer, stateless
- database.js — SQLite schema: users, characters, sessions ONLY — zero relationship/memory tables
- buildEncounterSystemPrompt — 11-section vessel-surrender prompt
- buildSceneSystemPrompt — wraps encounter prompt + scene/turn context

### Key Findings
1. ALL memory data is in-memory Maps — lost on restart
2. `crystallizeEncounter()` exists in PersonalityEvolutionService but is NEVER called by endEncounter/endScene
3. Database has zero tables for relationships, memories, encounters, or NPC evolution
4. Prompt builders already have slots for evolution/memory/opinions strings — the injection points exist
5. Two clean hooks for post-encounter synthesis: `endEncounter()` and `endScene()`

### What This Means for RelationshipRepository
- Need new SQLite tables for persistent relationship/memory storage
- Need automatic crystallization calls at end-of-encounter/scene
- Need to populate prompt builder inputs from persisted data on encounter start
- Existing services provide the in-session tracking; we just need the persistence layer + synthesis logic

## Session 3: Full NPC Name Path Trace — Identifying Display Name Leaks

### What Was Done
Traced the full path NPC names take from content data → DM engine → API routes → client rendering for both the encounter path (EncounterSessionService) and scene path (SceneEngine).

### Key Findings
1. **RelationshipRepository.getDisplayName() is only wired into NPC consciousness prompts** — it gates what the *NPC* perceives, not what the *player* sees.
2. **All player-facing fields use raw real names from content package** — `personality.name` / `participant.name` are used directly for:
   - `npcResponses[].senderName` (encounters)
   - `npcActions[].participantName` (scenes)
   - `transcript[].participantName` (scenes)
   - `npcs[].name` / `participants[].name` (both)
3. **API routes and controllers are pure passthroughs** — no name transformation happens between DM engine and client.
4. **SceneNarrator receives real names** and passes them to the LLM, which then uses them in narration text.
5. **The encounter path (EncounterSessionService.sendMessage) doesn't use RelationshipRepository at all** for name gating — no `nameResolver`, no `getDisplayName()` calls on the response path.

### Where Interception Must Happen
- Scene path: `SceneEngine.submitAction()` and `advanceNpcTurns()` where `npcActions` array and transcript entries are built, and where participants are serialized via `toJSON()`
- Encounter path: `EncounterSessionService.sendMessage()` where `senderName` is set, and `createEncounter()`/`getEncounter()` where `npcs[].name` is returned
- SceneNarrator: `narrateNpcBatch()` and `narrateSceneOpening()` where names are fed to the LLM
- SceneController: `createAtLocation()` where participant names are set from personality data

## Session 3: Free Will Rewrite — [HOW TO RESPOND] + [YOUR TURN]

### What Was Done
1. **Removed `[HOW TO RESPOND]` section entirely** from `buildEncounterSystemPrompt.js` (section 16). The encounter prompt now ends with the "Come in to focus" closing. No more commanding language ("Speak as yourself", "Stay in character", "Do not use markdown"), no formatting instructions, no restrictions.
2. **Replaced `[YOUR TURN]` section** in `buildSceneSystemPrompt.js` with free-will autonomy framing:
   - Removed `Round X. It is now your turn.` — no "turns" language
   - Reframed as "You are free to do whatever feels natural to you right now"
   - Kept all action types (SPEAK, ACT, OBSERVE, PASS, LEAVE) as possibilities, not commands
   - Kept [TO: name] targeting syntax and examples
   - Kept "no obligation to speak" and "do not invent events" guards
3. **Updated tests** across 3 test files:
   - `buildEncounterSystemPrompt.test.js` — removed "stay in character" and "markdown" assertions; replaced with negative assertions (does NOT contain commanding language)
   - `buildSceneSystemPrompt.test.js` — replaced "round number" test with "no turns language" test
   - `PromptContent.test.js` — updated req 7 to check for free-will framing instead of `[YOUR TURN]` header
4. **All 2774 tests passing**, 85 test files, zero failures.

### What Worked
- The philosophical distinction is clear: the encounter prompt surrenders the session to the consciousness, the scene prompt shows it the range of its autonomy. Neither commands.
- Keeping the action type brackets ([SPEAK], [ACT], etc.) as a parsing aid while framing them as choices, not instructions, strikes the right balance between free will and engine parsability.

### What Was Surprising
- The `[HOW TO RESPOND]` section had leaked into more tests than expected (markdown assertions, "not obligated" in minimal persona test). Thorough grep was essential.
- The "not obligated" sentiment moved naturally to the scene prompt's autonomy section — it fits there as part of "here's what you CAN do (including nothing)."

### What To Do Differently
- When removing a prompt section, always grep the entire test suite for phrases from that section — tests often assert exact wording.
- The `CharacterContextPackage.js` still has "Stay in character" in its own `[RESPONSE GUIDANCE]` section. This is a separate code path but may need the same philosophical treatment eventually.
