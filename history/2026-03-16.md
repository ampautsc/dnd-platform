## Key Decisions
- **Monorepo over multi-repo**: One developer, shared data contracts still being designed, atomic commits across boundaries
- **JavaScript/TypeScript (not Python)**: Existing production code is JS, no concrete reason to rewrite for API orchestration
- **TDD mandatory (Rule #4)**: No code without tests, period. Requirements → tests → implementation
- **Provider-agnostic AI**: LLM interface supports Claude, GPT, Gemini, local models
- **AI-generated images per scene**: Not pre-curated library
- **Private repo**: No restrictions that matter for this project

## Architecture Summary
- `content/` — Pure data, zero dependencies, every other package imports from it
- `combat/` — Crown jewel, immutable GameState, zero-trust TurnMenu
- `api/` — Database owner, auth (magic link), all persistent CRUD
- `gateway/` — WebSocket rooms, reconnection, voice signaling, no game logic
- `dm/` — The brain, story engine, NPC consciousness, narration, scene management
- `world/` — Background tick, NPC schedules, villain storylines, downtime
- `client/` — Thin mobile-first PWA, zero game logic

### What Worked
- TDD approach was efficient for UI — writing tests first caught text collision issues early (duplicate headings, overlapping button/status text)
- Demo fixtures in App.jsx make the site immediately testable without any backend running
- `useScreen` hook is simple state machine that avoids router library dependency
- Batch-creating all screen tests first, then all implementations, then fixing collisions was fast

### What Was Surprising or Went Wrong
- Text collision pattern: `/ready/i` matched both "✓ Ready" status text and "Ready Up" button. Same for `/guard/i` matching both NPC speaker and action button. Fix: use exact text or unique substrings.
- Duplicate heading: SessionLobby had `<h2>Session Lobby</h2>` and App wrapper had `<h1>Session Lobby</h1>`. Fix: removed h2 from component (parent provides heading).
- Playwright e2e file was being picked up by vitest — needed explicit `exclude: ['e2e/**']` in vite config.

### What I Learned
- **Pattern: parent owns headings, screens own content.** Avoids duplicate heading collisions when screens are composed in a router.
- **Pattern: exact text matchers for status indicators.** When a screen has buttons AND status text with similar words, use exact strings (`/✓ Ready/`) not loose patterns (`/ready/i`).
- **Pattern: screen state machine over router library.** For MVP with 6 screens and simple linear flow, `useState` + navigate function is simpler than importing react-router.
- **Pattern: demo fixtures for local dev.** Hardcoded mock data in App.jsx lets you test the full flow without backend. Replace with gateway events later.

### What To Do Differently Next Time
- Write all screen tests before any implementation (batch red phase across all screens) — caught more issues at the collision fixup stage but could have caught them earlier.
- Consider extracting heading into screen component as optional prop rather than removing it — that way the screen is usable standalone AND in a router.
