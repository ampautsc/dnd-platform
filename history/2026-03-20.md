**Results — 8/8 tests passed, 787ms total for 6 calls:**

| Scenario | Result | Time |
|----------|--------|------|

| Routine beer order ("I'll have an ale") | shouldReact: false, strength: 1 | 88ms |

**Groq free tier limits:**
- 30 RPM (handles 8 NPCs per utterance easily)
- 14,400 RPD (hours of active gameplay)
- 6,000 tokens/min
- No credit card required
- Model: llama-3.1-8b-instant (8B, much smarter than Phi-3.5-mini 3.8B)

**What worked:**
- Reusing the existing `openai` package with `baseURL` swap — zero new dependencies
- `response_format: { type: "json_object" }` produces clean JSON without grammar enforcement
- Terse system prompt (~6 lines) framed as "situations you INTERJECT into" rather than "topics you care about"
- `describe.skip` pattern when no API key — tests don't break CI

**What was surprising:**
- Groq is genuinely fast. 75ms for a classification call is faster than most database queries.
- llama-3.1-8b-instant on Groq comprehends nuance far better than Phi-3.5-mini on CPU — the "routine order vs insult" distinction was effortless.
- The reactionStrength values are much more decisive: 5 for clear reactions, 1 for clear non-reactions. No ambiguous 3s.

**What to do differently next time:**
- Start with cloud APIs for prototyping, then evaluate local models only if offline capability is a hard requirement
- The prompt framing as "situations you would INTERJECT into" vs "topics you care about" is a meaningful distinction — the former captures social judgment, the latter is just keyword matching by another name

**Decision: Jamba 1.5 Mini via Ollama — Rejected**
Researched extensively. Three independent reasons it doesn't work:
1. Jamba 1.5 Mini (52B params) is NOT in Ollama's registry
2. Requires ~32GB RAM quantized (all 52B MoE params must be in memory)
3. SSM O(n) advantage is irrelevant at 2000 tokens — bottleneck is model weight throughput, not attention complexity. From llama.cpp Mamba-2 developer: "comparable to Transformer models at 2k-4k context"

## Full 96-Scenario Groq Integration Test — Rate Limiting & Prompt Tuning

**Task:** Run the full 96 character-specific NPC reaction scenarios against the real Groq API to validate the entire ambient reaction pipeline end-to-end.

**What was done:**

### Run 1 — No throttle (baseline)
- 31/100 tests completed, 13 failed, 18 passed. 298s. Suite killed by timeout.
- Problem: No rate limiting. Groq free tier = 30 RPM + 6,000 TPM. Requests hit 429s, OpenAI SDK retried with backoff, most tests never completed.

### Fix 1 — Added RPM throttle to GroqReactionProvider
- Promise-queue based throttle in `_throttle()` method. Serializes concurrent requests with minimum interval between calls.
- New constructor option: `rpmLimit` (default 30 → later changed to 12).
- `evaluateReaction()` calls `await this._throttle()` before every API call.

### Run 2 — With 30 RPM throttle (2s intervals)
- 100/100 tests ran! But 52 failed, 48 passed. 1147s (19 min).
- **50 failures were `400: Failed to generate JSON`** — Groq's model couldn't produce valid JSON.
- Root cause: Prompt's JSON instruction was too weak ("Respond with JSON:") — got drowned out by long personality sections.

### Fix 2 — Strengthened JSON output instructions
- Added JSON preamble at START of system prompt: "You are an NPC reaction evaluator. You respond ONLY with JSON."
- Changed task instruction to include explicit example: `{"shouldReact": false, "reactionStrength": 1}`
- Bookend approach: JSON instruction at both start and end of prompt.

### Run 3 — Fixed JSON prompt, 30 RPM
- 100/100 ran. 39 failed, 61 passed. **0 JSON generation failures!** (down from 50). 1142s.
- All 39 failures were assertion errors (LLM wrong answers) — model was too trigger-happy (35 false positives).
- Still slow: 12-15s per test despite 2s throttle. Reason: hitting TPM limit (6,000 tokens/min).

### Fix 3 — Token-Per-Minute aware throttle + "Default to silence" prompt

## Prompt V4 — Fixed False Negatives for Direct-Address Utterances

**Task:** "Barkeep! Pour me your finest ale!" failed to trigger Samren (barkeep) or any NPC — even after softening the "default to silence" prompt. Iterated on `buildReactionPrompt.js` to fix.

**Root cause:** The single JSON example `{"shouldReact": false, "reactionStrength": 1}` anchored the 8B model toward silence. Combined with "stay quiet by default" and "clearly connects", the model treated every input as non-relevant.

**What was changed (v4 prompt):**
1. **Removed "stay quiet by default"** — replaced neutral framing: "Would you respond?"
2. **YES conditions listed first** — model reads top-down, anchors to first instructions
3. **Explicit role keywords** — "barkeep, innkeeper, waitress, etc." in the "addresses you by role" criterion
4. **Two JSON examples** — both `true` and `false` shown, de-anchoring from silence
5. **Removed "ONLY" and "clearly"** — lowered the reaction threshold to natural level

**Test results with v4 prompt:**
| Utterance | Result | Correct? |
|---|---|---|
| "Barkeep! Pour me your finest ale!" | Samren 4/5, Carza 2/5, Clifton 3/5 | YES |
| "Someone swindled me out of 50 gold!" | Carza 5/5, Samren 5/5, Woody 5/5 | YES |
| "Old garrison tower is haunted..." | Clifton 4/5 | YES |
| "Weather's been pleasant" | Nobody | YES |
| 67 CI-safe mocked tests | ALL PASSING | YES |

**What worked:**
- The JSON example anchoring theory was correct — showing only `false` as the example was the primary cause of silence bias on a small model.
- Listing YES conditions first (before NO conditions) gives the model permission to react.
- Explicit role keywords ("barkeep, innkeeper, waitress") bridge the gap between the utterance term and the NPC's identity.

**What was surprising:**
- The "softened v3" prompt (removing "Most things are none of your business" but keeping "stay quiet by default") made zero difference. The JSON example anchor was the dominant factor.
- A single false example `{"shouldReact": false, ...}` is a stronger signal than all the text instructions combined for an 8B model.

**What to do differently next time:**
- For LLM JSON classifiers: ALWAYS show both positive and negative examples. Never show only the default/negative case.
- For small models (8B): instruction ordering matters more than wording. Put desired behavior first.
- Test prompt changes with direct-address cases first — they're the easiest positive case and detect anchor bias instantly.
