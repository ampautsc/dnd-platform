## Session 1: Azure Deployment Workflows

### What was done
Added 4 GitHub Actions workflows for Azure deployment:
- `azure-provision.yml` — one-time provisioning workflow (creates Resource Group, App Service Plan, two App Services, Static Web App, wires GitHub secrets)
- `deploy-client.yml` — deploys `packages/client/` to Azure Static Web Apps
- `deploy-api.yml` — deploys `packages/api/` + dm/content/combat/world to Azure App Service
- `deploy-gateway.yml` — deploys `packages/gateway/` to Azure App Service

Also added `AZURE_SETUP.md` with full setup instructions (SP creation, PAT, running provision workflow).

CodeQL scan flagged missing permissions blocks → fixed with explicit `permissions: contents: read` on all jobs.

### Why this was needed
User wants to host the platform on Azure. The provision workflow automates the full Azure setup — no manual portal clicking needed.

### Why merge was required
`workflow_dispatch` triggers only appear in GitHub's Actions UI when the workflow file is on the **default branch** (master). The workflows were on the PR branch — invisible in Actions until merged.

### What worked
- `azure/login@v2` + `az` CLI for all resource creation
- `gh` CLI (via GH_PAT secret) to write GitHub secrets/variables automatically from within the workflow
- Graceful degradation: if a resource already exists, the workflow continues rather than failing

### What the user was frustrated about
Merging the PR closes the Copilot session. New sessions don't have context. User was rightfully upset about losing context. Solution: created `docs/handoffs/2026-04-04-post-azure-merge.md` as a full context dump for the next session.
